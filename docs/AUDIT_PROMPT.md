# Промпт для ручной проверки заданий (LLM / эксперт)

Используйте этот промпт с любым LLM или как чек-лист при ручном аудите файла `client/src/lib/taskData.ts` (задания Buddy VPR, русский язык, ВПР-4).

---

## Задача

Ты — аудитор контента. Проверь массив заданий `tasksData` в TypeScript-файле на:

1. **Структуру**
   - У каждого задания есть поля: `id`, `type`, `word`, `correct`, `difficulty`, `category`.
   - `difficulty` только 1 или 2.
   - `category` и `type` из допустимого набора: accent, phonetics, meaning, morphemics, morphology, syntax, reading, plan, vocabulary, context.

2. **Согласованность**
   - Поле `ruleId` (если есть) должно соответствовать правилу из `goldenRules` с тем же `id`.
   - Категория задания должна быть согласована с категорией правила (ruleId → category).

3. **Текстовые задания (inputType: "text")**
   - Должны содержать непустой массив `acceptableAnswers` для fuzzy-валидации.
   - Желательно 2–5 элементов в `keywords` для семантической проверки ответа.
   - Массив `unacceptablePatterns` (если есть) — для этического фильтра; формулировки не должны поощрять пассивность или нежелательное поведение.

4. **Качество формулировок**
   - Вопросы и подсказки без опечаток, в нейтральном и поддерживающем тоне.
   - Для заданий с `passage` и `instruction`: инструкция короткая (до ~140 символов), вопрос — отдельно и чётко сформулирован.

5. **Дублирование контента (по желанию)**
   - Проверь, не повторяется ли один и тот же текст в полях `question`, `correct`, `hint` (полное или почти полное совпадение). Если да — предложи развести формулировки.

6. **Орфография в эталонных ответах (по желанию)**
   - В `correct` и `acceptableAnswers` не должно быть орфографических ошибок. При необходимости предложи правки или пометку «проверить по словарю».

---

## Формат ответа

Для каждого найденного нарушения укажи:
- **Где:** ID задания (и при необходимости поле).
- **Что не так:** краткое описание.
- **Рекомендация:** как исправить.

В конце дай общий вывод: аудит пройден / пройден с замечаниями / не пройден.

---

## Расширения (по желанию)

- **Проверка на дублирование текста:** fuzzy match между `question`, `correct`, `hint` — выделить задания, где эти поля слишком похожи (риск подсказки ответа в вопросе или подсказке).
- **Орфографический словарь:** при валидации или аудите проверять `acceptableAnswers` по словарю (например, подключить API или локальный словарь), помечать сомнительные варианты.

## Интеграция в CI/CD (опционально)

В GitHub Actions можно добавить шаг проверки перед merge:

```yaml
# .github/workflows/audit.yml (или шаг в существующем workflow)
- name: Audit task data
  run: |
    python3 scripts/audit_tasks.py
    if [ $? -ne 0 ]; then
      echo "❌ Audit failed. Fix errors before merge."
      exit 1
    fi
```

Запуск локально из корня репозитория:

```bash
python3 scripts/audit_tasks.py
# или с указанием файла:
TASK_FILE=client/src/lib/taskData.ts python3 scripts/audit_tasks.py
```
