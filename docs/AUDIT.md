# Аудит приложения ВПР Бадди

*Проведён по состоянию на текущую дату. Для плана День 2 (аудит PWA) и общей оценки.*

---

## Что сделано хорошо

### Архитектура и код
- **Чёткое разделение:** client / server / shared — удобно поддерживать.
- **Типизация:** TypeScript, Drizzle-схема, общие типы в `shared/schema.ts`.
- **API:** маршруты логично сгруппированы, есть валидация (zod) на критичных эндпоинтах (например, `roundQuerySchema` для `sessionId`).
- **Fallback данных:** при отсутствии ответа API используются локальные `tasksData` и `goldenRules` — приложение не падает без сети.
- **Error Boundary:** добавлен — при ошибке React показывается сообщение вместо белого экрана.
- **Семантика и темы:** русский интерфейс, «мягкая» палитра (mint, pastel), без агрессивного красного для ошибок — соответствует аудитории 7–14 лет.

### Контент и геймификация
- **381 задание**, 27 правил в сиде — хорошая база для русского 4 класса.
- **Раунды и мастерство:** round-based повторение, «Освоено!», список ошибочных слов — продуманный сценарий.
- **Звёзды, острова, прогресс:** визуально и по логике соответствует задумке «тренажёр с напарником».

### PWA и офлайн
- **manifest.json:** имя, short_name, theme_color, иконки, display: standalone — настроен.
- **Service Worker (sw.js):** есть; кэширует `/`, `/manifest.json`, `/favicon.png`; для `/api/*` при офлайне возвращает `{ error: "offline" }`; для остальных GET — network-first с кэшем при успехе.
- **Регистрация SW:** в `main.tsx` регистрируется с `.catch(() => {})` — не ломает загрузку при ошибке.

### Доступность и mobile-first
- **Шрифт:** Nunito, крупный — подходит для детей.
- **Кнопки:** в ряде мест используются размеры вроде `w-11 h-11`, `min-h-9`; часть интерактивных элементов может быть чуть меньше 44px на мобильных — стоит проверить по макету (рекомендация: минимум 44×44 для touch).
- **Цвета:** не только «красный = ошибка» — учтено в описании продукта.

## Что доработать или учесть

### 1. Офлайн и данные (под план «гибрид»)
- **Задания в офлайне:** SW не кэширует `/src/*` и не кэширует ответы API. При полном офлайне после первой загрузки список заданий с сервера недоступен; выручает только fallback на локальные `tasksData` в коде. Для настоящего офлайн-first по плану нужны: кэш заданий (например, в IndexedDB) и/или предзагрузка списка заданий в SW/кеш при первом визите.
- **Прогресс:** сейчас прогресс уходит на сервер (session, round, answer). В офлайне нужна очередь (localStorage/IndexedDB) и синхронизация при появлении сети — в коде пока нет.

### 2. Безопасность
- **Таблица `users` в схеме:** пароль в открытом виде (`text("password")`). Если позже появится авторизация (родительский дашборд и т.п.) — пароли нужно хранить только в виде хэша (bcrypt/argon2), не в plain text.
- **`.env`:** добавлен в `.gitignore` — секреты не попадут в репозиторий при коммите.
- **Данные детей:** персональные данные не собираются; только sessionId и прогресс. Для 152-ФЗ при масштабировании важно явно описать, что хранится и где (документация/политика).

### 3. PWA и кэш
- **Объём кэша SW:** кэшируются только три статических ресурса. HTML и JS идут через Vite dev server; в production после `npm run build` нужно убедиться, что SW кэширует собранные JS/CSS и index.html (или что стратегия кэша явно определена в скрипте сборки).
- **Версионирование:** `CACHE_NAME = "vpr-buddy-v1"` — при смене версии старый кэш очищается; хорошо. Имеет смысл привязать версию к версии приложения или дате сборки.
- **Иконки PWA:** в manifest указаны 192 и 512 с одним и тем же `/favicon.png`. Лучше иметь отдельные файлы под 192 и 512 для «Добавить на экран» и splash.

### 4. Надёжность и UX
- **Запросы к API:** при ошибке (например, 500) `throwIfResNotOk` бросает исключение; запрос падает, но есть fallback на локальные данные и Error Boundary — приемлемо. Для лучшего UX можно показывать тост «Нет связи, работаем с сохранёнными данными».
- **Индикатор офлайна:** в интерфейсе нет явного «Вы в офлайне» — по плану День 5 это стоит добавить.

### 5. Производительность
- **Первая загрузка:** шрифт Nunito тянется с Google Fonts — при плохом интернете может задерживать отрисовку. Рассмотреть self-host шрифта или `font-display: optional` для ускорения FCP.
- **Размер бандла:** много UI-компонентов (radix и т.д.); при росте приложения имеет смысл проверить code-splitting по маршрутам/экранам.

### 6. Релиз и хостинг
- **Сервер:** Express слушает `127.0.0.1` — для локальной разработки верно. Для деплоя (Vercel/Netlify и т.д.) нужен либо serverless-обход (отдельный backend на Node/другом стеке), либо хостинг с длительным процессом (например, Yandex Cloud), т.к. текущий стек — один процесс Express + Vite в dev.
- **База:** сейчас локальный PostgreSQL; для продакшена понадобится облачная БД (уже обсуждали Supabase/Neon/РФ-хостинг).

## Соответствие целям проекта

| Требование            | Статус |
|-----------------------|--------|
| Офлайн-режим          | Частично: SW есть, fallback по данным есть; полноценный кэш заданий и синхронизация прогресса — в плане |
| Mobile-first          | Да: адаптивная вёрстка, крупные элементы |
| Геймификация          | Да: острова, звёзды, раунды, мастерство |
| Бадди-персонаж        | Да: Mascot, подсказки, тосты |
| Без «красного» для ошибок | Да |
| 152-ФЗ / персданные   | Пока не собираются; при появлении авторизации — нужна политика и хэш паролей |

## Рекомендации по приоритету

1. **Короткий срок:** проверить touch-цели (кнопки/карточки ≥ 44px), при необходимости подправить классы.
2. **План День 5 (MVP офлайн):** добавить кэш заданий (IndexedDB или расширенный кэш SW), индикатор «нет интернета», при желании — очередь прогресса и синхронизацию.
3. **Перед продакшеном:** хэш паролей при использовании таблицы `users`; явная стратегия кэша для production build; self-host или оптимизация загрузки шрифта.
4. **Документация:** кратко зафиксировать, какие данные хранятся (sessionId, прогресс по категориям), где и зачем — для 152-ФЗ и доверия родителей.
