# Аудит: домен buddyvpr.ru и ре-деплой на Vercel

**Цель:** чтобы https://www.buddyvpr.ru открывался с зелёным замком во всех браузерах.

---

## Шаг 1: Диагностика (почему Pending / не открывается)

### 1.1 Какие домены должны быть в проекте

В проекте Vercel должны быть привязаны **оба**:

| Домен            | Назначение |
|------------------|------------|
| **buddyvpr.ru**  | Корневой; обычно редирект на www |
| **www.buddyvpr.ru** | Основной Production-домен |

Пока **buddyvpr.vercel.app** открывается — приложение и билд в порядке. Проблема именно в цепочке: DNS → Vercel → SSL.

### 1.2 Почему документ (doc) в статусе Pending в Chrome

**Возможные причины:**

1. **SSL-сертификат ещё не готов**
   - После добавления домена или смены DNS Vercel (Let's Encrypt) запрашивает сертификат. Пока он выдаётся (1–15 минут, редко дольше), браузер может держать запрос в **Pending** или получить таймаут.
   - **Что делать:** подождать 10–15 минут после появления домена в Vercel и проверки DNS, затем обновить страницу с отключённым кэшем.

2. **Редиректы www ↔ non-www**
   - Если в Vercel настроен редирект (например, buddyvpr.ru → www.buddyvpr.ru), а DNS для одного из вариантов указывает не туда или ещё не обновился — браузер может уйти в цикл или висеть в Pending.
   - **Что делать:** в Vercel в **Settings → Domains** для каждого домена должен быть явно выбран тип: **Redirect** (на другой домен) или **Production**. Не должно быть двух доменов в роли Production с разным поведением без явного редиректа.

3. **DNS ещё не обновился у тебя / у провайдера**
   - После правок на nic.ru распространение DNS — от нескольких минут до 24–48 часов. Если у тебя или у Vercel ещё старая запись — соединение идёт не на Vercel, и документ висит.
   - **Что делать:** проверить DNS через `dig www.buddyvpr.ru CNAME` или сервис dnschecker.org — значение должно совпадать с тем, что указано в Vercel (Settings → Domains) для www, например `*.vercel-dns-017.com` или `cname.vercel-dns.com`.

4. **Кэш браузера / CDN**
   - Старый ответ (ошибка или редирект) может кэшироваться. Тогда кажется, что «ничего не изменилось».
   - **Что делать:** DevTools → Network → **Disable cache**, затем жёсткое обновление (Cmd+Shift+R / Ctrl+Shift+R). Попробовать в режиме инкогнито.

### 1.3 Чек-лист в Vercel (Settings → Domains)

Сделай по порядку:

1. Зайди в **Vercel** → проект **Buddy-VPR** → **Settings** → **Domains**.
2. Убедись, что в списке есть:
   - **buddyvpr.ru**
   - **www.buddyvpr.ru**
3. Для каждого домена проверь статус:
   - **Valid Configuration** — всё ок, осталось дождаться SSL и проверить в браузере.
   - **DNS Change Recommended** / **Invalid Configuration** — на nic.ru нужно выставить **ровно те значения**, что Vercel показывает в блоке «Vercel recommends updating your DNS records» (см. таблицу ниже).
4. Если статус был Invalid и ты поправил DNS — в Vercel нажми **Refresh** (или «Verify») рядом с доменом и подожди 5–10 минут.
5. **Одним кликом:** кнопка **Refresh** / **Verify** в строке домена как раз и перепроверяет конфигурацию; исправить саму ошибку «одним кликом» нельзя — правки только в DNS у регистратора (nic.ru).

Итог: цель — увидеть для **buddyvpr.ru** и **www.buddyvpr.ru** статус **Valid Configuration**. Тогда SSL и открытие сайта должны заработать после ожидания 5–15 минут.

### 1.4 Записи для nic.ru (брать из Vercel → Domains)

**Важно:** Vercel может показывать **другие** значения, чем в старых инструкциях (новый IP, проект-специфичный CNAME). Всегда бери актуальные из **Settings → Domains** в твоём проекте.

По скриншоту Vercel (февраль 2026) для проекта Buddy-VPR нужны такие записи в **Управление зоной** в nic.ru:

| Тип     | Имя (поддомен) | Значение |
|--------|-----------------|----------|
| **A**  | `@`             | **216.198.79.1** |
| **CNAME** | **www**      | **41d5deb2dfca553c.vercel-dns-017.com** |

- Для корня **buddyvpr.ru**: одна A-запись с именем `@` и значением **216.198.79.1** (старый IP 76.76.21.21 Vercel тоже поддерживает, но рекомендует новый).
- Для **www**: CNAME с именем **www** и значением **41d5deb2dfca553c.vercel-dns-017.com** (в nic.ru точку в конце добавляй только если интерфейс этого требует).
- Если в панели Vercel у тебя указаны **другие** значения — вводи в nic.ru именно их. После сохранения зоны нажми в Vercel **Refresh** у домена и подожди 5–15 минут.

## Шаг 2: Принудительный ре-деплой (Redeploy)

### 2.1 Через панель Vercel

1. **Vercel** → проект **Buddy-VPR** → вкладка **Deployments**.
2. У последнего успешного деплоя (статус **Ready**) открой меню (**три точки** ⋮).
3. Выбери **Redeploy**.
4. В диалоге:
   - **Рекомендация:** сними галочку **Use existing Build Cache**. Так Vercel пересоберёт проект с нуля и гарантированно подхватит актуальные настройки (в т.ч. домены/конфиги). Для «оживления» домена после смены DNS это предпочтительно.
5. Подтверди **Redeploy**. Дождись статуса **Ready** (обычно 1–3 минуты).

### 2.2 Через CLI (если установлен Vercel CLI)

Из корня проекта:

```bash
cd /Users/nathalie/Buddy-VPR
vercel --prod
```

Это создаёт новый production-деплой (как после push). Кэш при этом зависит от настроек проекта; при необходимости полной пересборки в панели всё равно можно сделать Redeploy без кэша, как выше.

## Шаг 3: Что проверить после того, как сайт откроется

Через 10–15 минут после того, как в Vercel у доменов станет **Valid Configuration** и деплой будет **Ready**, проверь по пунктам:

| № | Проверка |
|---|----------|
| 1 | **HTTPS и замок:** открывается ли **https://www.buddyvpr.ru** с зелёным замком (без предупреждений о сертификате)? |
| 2 | **Редирект:** при вводе **http://www.buddyvpr.ru** или **https://buddyvpr.ru** (без www) происходит ли переход на **https://www.buddyvpr.ru** без ошибок? |
| 3 | **Нет Mixed Content:** в DevTools → Console нет ли ошибок вида «Mixed Content» (запрос по HTTP с HTTPS-страницы)? |
| 4 | **Главная и навигация:** загружается ли главный экран приложения, переключение по островам/режимам работает? |
| 5 | **Мобильный:** открывается ли **https://www.buddyvpr.ru** в Safari (iOS) и в Chrome (Android) с замком и без предупреждений? |

Если все пять пунктов ок — инфраструктура для основного домена настроена корректно.

## Кратко

- **Почему Pending:** чаще всего SSL ещё выдаётся после смены DNS, реже — некорректные редиректы www/non-www или старый DNS/кэш.
- **Что сделать в Vercel:** **Settings → Domains** — оба домена в статусе **Valid Configuration**; при смене DNS нажать **Refresh** и подождать 5–15 минут.
- **Redeploy:** **Deployments** → ⋮ у последнего деплоя → **Redeploy** без галочки **Use existing Build Cache**; альтернатива — `vercel --prod` из корня проекта.
- **После оживления домена:** чек-лист выше (HTTPS, редирект, нет Mixed Content, работа приложения, проверка на мобильных).
