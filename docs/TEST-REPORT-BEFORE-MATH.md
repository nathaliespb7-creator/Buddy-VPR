# Отчёт: комплексное тестирование Buddy-VPR перед добавлением модуля Математики

**Ветка:** clean-start  
**Дата:** 2026  
**Цель:** убедиться, что архитектура и логика готовы к расширению (Математика, Окружающий мир).

---

## 1. Анализ архитектуры и кода

### 1.1 Общая структура

- **Роутинг:** wouter, цепочка экранов: `/` (Welcome) → `/intro` (Landing) → `/class-selector` (выбор предмета) → `/play` (Home, карта островов).
- **Данные:** основной контент заданий в `client/src/lib/taskData.ts` (один массив `tasksData`), категории совпадают с ключами островов в `IslandMap.tsx`.
- **API:** `queryClient.ts` — единая точка (API_BASE, fetchWithTimeout, apiRequest, getQueryFn). При пустом/неверном API_BASE запросы идут на тот же origin (Vercel отдаёт HTML → fallback на локальные данные).

### 1.2 queryClient.ts и хранение данных

| Аспект | Состояние |
|--------|-----------|
| **API_BASE** | Нормализация есть: пусто, `/api`, `/api/` не дублируются в путь. |
| **Таймаут** | 15 с для fetch — ок. |
| **Ошибки** | `throwIfResNotOk` бросает при !res.ok; запросы с кастомным `queryFn` (tasks, progress в IslandMap) при ошибке возвращают `[]` и не роняют приложение. |
| **Риск** | В `Home.tsx` один запрос **без** кастомного `queryFn`: `useQuery({ queryKey: ["/api/categories/progress?sessionId=..."] })`. Используется глобальный `getQueryFn`, который при 404/500 бросает. TanStack Query сохраняет ошибку в состоянии; компонент читает только `data`, поэтому краша нет, но при включении useErrorBoundary или отображении ошибок пользователь может увидеть падение. Рекомендация: задать этому запросу такой же кастомный `queryFn` с возвратом `[]` при ошибке. |

### 1.3 localStorage

| Ключ | Где используется | Разделение по предметам |
|------|------------------|--------------------------|
| `buddy_session_id` | Home.tsx | Нет — один на всё приложение. |
| `buddy_profile` | Home.tsx (звёзды, аватар, tier) | Нет — один профиль, звёзды не привязаны к предмету/классу. |
| `buddy_settings_v1` | SettingsContext (анимация, выбор уровня) | Нет, и не требуется. |

**Вывод:** данные **не** разделены по модулям/предметам. Добавление Математики потребует решения: один общий профиль с разбивкой звёзд по предметам или отдельные ключи/структура (например `buddy_profile_math`, или в одном объекте `starsBySubject`).

### 1.4 Спагетти и зависимости

- **Home.tsx** — большой файл (много фаз, таймеры, мутации, острова). Логика местами плотная, но не «спагетти»; зависимости от taskData/IslandMap явные.
- Жёстких циклических зависимостей не видно.
- **Дублирование конфигурации островов:** в `subjectsAndClasses.ts` (русский: ключи + подписи) и в `IslandMap.tsx` — полный массив `islands` (ключи, подписи, иконки, цвета). При добавлении Математики придётся либо вести один источник правды (например конфиг по предметам), либо дублировать ещё один набор островов в IslandMap.

### 1.5 Хардкод, мешающий масштабированию

| Место | Что захардкожено | Влияние на Математику |
|-------|-------------------|------------------------|
| **IslandMap.tsx** | Массив `islands` (11 островов русского языка) | Для Математики нужен другой набор островов или параметр `subjectId` и выбор конфига. |
| **Home.tsx** `startDiagnostic` | Список категорий: `["accent", "phonetics", "meaning", "morphemics", "morphology", "syntax", "reading", "plan", "vocabulary"]` | Только русский. Для математики нужны свои категории или конфиг по предмету. |
| **MixedModeChoiceScreen** | `CATEGORY_ORDER = ["accent", "phonetics", "meaning", "morphemics", "morphology", "syntax"]` | То же — привязка к русскому. |
| **taskData.ts** | Один массив `tasksData`, все задания с `category` из набора русского языка | Математика = отдельный набор заданий и категорий (или общий интерфейс + разная загрузка по предмету). |
| **ClassSubjectSelector** | Выбранный предмет не передаётся в `/play` (ни URL, ни контекст) | На `/play` сейчас всегда «русский» контент. Чтобы играть по математике, нужно передать subject (и при необходимости класс) и подставлять соответствующие задания/острова. |

## 2. Тестология логики и навигации

### 2.1 Полный путь пользователя (симуляция по коду)

| Шаг | Ожидание | Факт по коду |
|-----|----------|--------------|
| Главная `/` | Экран с маскотом, кнопка «Начать» | WelcomePage рендерит SplashScreen, по «Начать» → `setLocation("/intro")`. |
| Описание `/intro` | Текст про тренажёр, кнопка «Выбери класс» | LandingPage, кнопка → `setLocation("/class-selector")`. |
| Выбор предмета `/class-selector` | Выпадающий список предметов, кнопка «Старт» | ClassSubjectSelector: только предмет (класс убран), «Старт» → `setLocation("/play")`. **Выбранный предмет никуда не передаётся.** |
| Игра `/play` | Карта островов, задания | Home загружает `tasksData` (русский) и острова из IslandMap (русский). Выбор предмета на предыдущем шаге не используется. |

**Итог:** навигация цепочки работает; передача «выбранный предмет/класс» в игру не реализована — на `/play` всегда один и тот же контент (русский).

### 2.2 Сохранение и передача данных

- **Профиль и звёзды:** сохраняются в `localStorage` при действиях в Home; при F5 подхватываются из `getStoredProfile()`.
- **Выбор предмета на ClassSubjectSelector:** не сохраняется (не в URL, не в localStorage, не в контексте). После F5 на `/class-selector` снова дефолтный предмет.
- **Сессия:** `buddy_session_id` создаётся/читается из localStorage, при F5 сохраняется.

### 2.3 Перезагрузка (F5)

- **На `/`, `/intro`, `/class-selector`:** только UI; потери данных нет (данных с этих экранов и не сохраняем).
- **На `/play`:** профиль и sessionId восстанавливаются из localStorage; прогресс по раундам при отсутствии бэкенда не сохраняется между сессиями (только локальные звёзды).

### 2.4 Поведение при ошибке (нет сети / API недоступен)

- **Загрузка списка заданий** (Home): кастомный `queryFn` при ошибке возвращает `[]`, затем используется fallback `tasksData` — приложение не падает, задания есть.
- **Загрузка прогресса по категориям** (IslandMap): кастомный `queryFn` при ошибке возвращает `[]` — прогресс пустой, карта отображается.
- **Загрузка прогресса по категориям** (Home, общий прогресс): используется **дефолтный** queryFn — при 404/500 запрос в состоянии error, `categoryProgressData` = undefined, `overallProgress` = 0. Краша нет.
- **Загрузка раунда** (остров): в `handleSelectIsland` при ошибке fetch в catch берутся локальные задания по категории и стартует тренировка — пользователь может продолжать.
- **ErrorBoundary:** ловит ошибки рендера и показывает «Что-то пошло не так» + кнопку «Попробовать снова». Сетевые ошибки в useQuery не пробрасываются в boundary (если не включён useErrorBoundary).

**Вывод:** при отсутствии интернета/API приложение не падает; показывается локальный контент. Явного сообщения вида «работаем офлайн» нет — можно добавить для UX.

## 3. UI/UX и визуальная проверка

Проверка по коду и типовым паттернам:

- **Адаптивность:** используются классы Tailwind (sm:, max-w-, min-h-[48px], safe-bottom и т.д.), единицы 100dvh; кнопки и зоны нажатия выглядят рассчитанными на тач.
- **Тексты:** на второй странице (LandingPage) подпись под кнопкой: «Шаг 1: выбери класс и предмет» — на третьем экране сейчас только предмет (класс убран). Стоит заменить на «Выбери предмет — дальше тебя проведёт Бадди» или аналогично.
- **Цвета:** emerald/teal/sky в градиентах, кнопки primary; контраст по умолчанию достаточный (slate-900 на светлом фоне). Брендбук в репозитории не просматривался — явных расхождений не видно.

## 4. Поиск багов и ошибок

### 4.1 TypeScript

- Выполнено: `npx tsc --noEmit` — **без ошибок** (exit code 0). Предупреждений в выводе нет.

### 4.2 Потенциальные проблемы по коду

| Проблема | Критичность | Где |
|----------|-------------|-----|
| Запрос `/api/categories/progress` в Home без кастомного queryFn — при падении API запрос в error, данные undefined; при будущем useErrorBoundary возможен краш. | Важная | Home.tsx ~340 |
| Выбранный предмет (и класс) не передаётся в /play — игнорируется выбор пользователя. | Важная (для мультипредметности) | ClassSubjectSelector → Home |
| Подпись «выбери класс и предмет» при только выборе предмета. | Мелкая | LandingPage.tsx |
| В taskData есть вызовы вида `tasksData.find(t => t.id === 1)!` — при изменении id или порядка возможен runtime-баг. | Мелкая | taskData.ts |

### 4.3 Консоль браузера (ожидаемо при front-only деплое)

- **404:** возможны на `/api/tasks`, `/api/categories/progress`, `/api/round/...` если бэкенд не поднят. Обработано fallback’ами и try/catch.
- **undefined/null:** в коде есть проверки (например `profile?.stars`, optional chaining); места с возможным undefined приходят из уже обработанных запросов (data может быть undefined — это учитывается).

### 4.4 LocalStorage

- Чтение/запись обёрнуты в try/catch в SettingsContext; в Home при getStoredProfile — try/catch. При переполнении или приватном режиме запись может не сработать — падения не будет, но настройки/профиль не сохранятся. Для MVP приемлемо.

## 5. Оценка готовности к расширению (Математика)

### Насколько легко добавить новый предмет?

**Сейчас — не легко.** Потребуются целенаправленные изменения:

1. **Контент:** отдельный массив заданий для математики (по аналогии с `tasksData`) или общий формат с полем `subjectId`/`category` и загрузка по предмету.
2. **Острова:** либо второй массив островов в IslandMap с переключением по предмету, либо вынести конфиг островов в данные (например из subjectsAndClasses или отдельный модуль) и рендерить по выбранному предмету.
3. **Передача выбора:** с `/class-selector` передавать в `/play` subjectId (и при необходимости class) через searchParams (wouter) или контекст, и в Home читать и подставлять соответствующие задания и острова.
4. **Профиль/прогресс:** решить, хранить ли звёзды и прогресс в одном профиле с разбивкой по предметам (например `starsBySubject`) или отдельные ключи — и отразить это в типах и в API (если будет).

### Потенциальные проблемы при добавлении Математики

- Дублирование логики диагностики/режимов (startDiagnostic, MixedModeChoiceScreen) под второй предмет или рефакторинг в предмето-независимый вид (конфиг категорий по предмету).
- Разные форматы заданий (русский: слово/варианты/подсказки; математика может быть числа/формулы) — возможна необходимость расширения типа Task или отдельного типа для математики с общей оболочкой.
- API прогресса/раундов: если бэкенд один, нужно согласовать формат (sessionId + subjectId или отдельные эндпоинты по предметам).

## 6. Финальный отчёт

### Баги по приоритету

| Уровень | Описание |
|---------|----------|
| **Критичные** | Не выявлены. При отсутствии API приложение не падает, есть fallback на локальные задания. |
| **Важные** | (1) В Home один useQuery прогресса без кастомного queryFn — при падении API возможен error state; лучше унифицировать с возвратом `[]`. (2) Выбор предмета на ClassSubjectSelector не используется на `/play` — при добавлении Математики это обязательно нужно исправить. |
| **Мелкие** | (1) Текст «выбери класс и предмет» на втором экране при только выборе предмета. (2) Хардкод списка категорий в startDiagnostic и MixedModeChoiceScreen — усложнит добавление второго предмета. |

### Рекомендации по архитектуре перед следующим этапом

1. **Единый источник островов:** вынести конфиг островов (ключ, лейбл, иконка, цвета) в данные (например по subjectId) и использовать в IslandMap и при диагностике, чтобы не дублировать списки категорий в нескольких файлах.
2. **Передача subject (и при необходимости class) в /play:** через searchParams (`/play?subject=math`) или React-контекст, инициализируемый с ClassSubjectSelector, чтобы Home и IslandMap подставляли контент по выбранному предмету.
3. **Профиль и прогресс:** заложить структуру «по предметам» (например звёзды/прогресс по subjectId) в профиле и в API, чтобы при добавлении Математики не переделывать формат хранения.
4. **Обработка ошибок API:** для запроса прогресса в Home добавить кастомный queryFn с возвратом `[]` при ошибке и при желании показывать пользователю короткое сообщение об офлайн-режиме, когда используется только локальный контент.

### Готовность к добавлению Математики

**Архитектура и логика в текущем виде работают стабильно:** критичных багов нет, навигация и офлайн-fallback согласованы. К добавлению Математики проект **готов в том смысле, что не мешают критические ошибки**, но **потребуется доработка**: передача выбора предмета в игру, разделение контента и островов по предметам, и решение по хранению прогресса/звёзд по предметам. Рекомендуется сначала внедрить пункты 1–2 из рекомендаций (и при необходимости пункт 3), затем добавлять контент и модуль Математики.
